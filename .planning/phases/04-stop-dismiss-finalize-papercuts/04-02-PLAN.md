## Phase Goal

**As a** human operator or AI agent managing an automated guarded rollout that has detected a metric regression, **I want to** dismiss the regression in one CLI invocation that pre-reads state, refuses if there is no active regression, sends the dismissRegression instruction, and waits with bounded backoff until the post-dismiss state is reflected before returning, **so that** I can resume the rollout (or surface a definitive timeout-warning when the dismissal hasn't propagated) without ever needing the LaunchDarkly UI and without baking polling intelligence into every downstream consumer.

<objective>
Ship `ldcli flags rollouts-beta dismiss-regression` end-to-end as a thin vertical slice — command body, typed Client.DismissRegression method (PATCH semantic-patch + bounded-backoff re-fetch loop because PC-007 means the 204 response carries no state), flesh out DismissRegressionInstruction, mock parity, one new error-code constant, plaintext rendering, JSON envelope with `meta.uiURL`, full test coverage including the eventual-consistency timeout branch.

Purpose: deliver Phase 4 SC#2 (dismiss-regression with re-fetch+backoff), SC#3 dismiss subset (no-active-regression + no-rollouts agent-friendly errors), and SC#4 dismiss subset (confirmation envelope + meta.uiURL). Reuses the Plan 04-01 substrate verbatim (`BuildUIURL`, `NewRolloutEnvelopeWithUI`, `ErrCodeAlreadyTerminal` is NOT reused here but its sibling pattern is); adds one new typed client method, one new instruction shape, one new error-code constant, and one new subcommand.

Output: a working `dismiss-regression` subcommand. Sets up Plan 04-03 (real-staging smoke) to exercise BOTH stop AND dismiss against staging, and Plan 04-04 (DOC-03 + LEARN-03) to close out the milestone.

**Prototype-first framing carry-forward (memory `project_prototype_first_framing.md`):** This is the last code-shipping plan of the milestone. The CLI is a learnings-vehicle for the API team; ship working code, capture papercuts at every workaround site, and DO NOT invent envelope normalizations to paper over the 204 — the bounded-backoff loop is a UX affordance (mirrors how a human operator would refresh the UI), but the JSON envelope's `data` is the verbatim API-passthrough of the re-fetched Rollout. The polling budget itself is hard-coded per RESEARCH.md architecture guidance (~10s timeout, 1s/3s backoff); making it configurable is out of scope and would create a CLI/UX surface (a `--poll-timeout` flag) the production CLI build can re-litigate.
</objective>
